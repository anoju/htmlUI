Markdown은 복잡한 편집기 없이도 제목, 목록, 표, 코드, 링크, 이미지가 포함된 문서를 빠르게 작성하는 문법입니다. 이 문서는 실제 .md 파일에 그대로 붙여 넣을 수 있는 예시 중심으로 구성했습니다.

1. Markdown이란?

Markdown은 일반 텍스트에 약속된 기호를 붙여 문서 구조를 표현하는 가벼운 마크업 언어입니다. 파일 확장자는 보통 .md 또는 .markdown을 사용합니다.

왜 사용하나요?

  • 메모장 같은 단순 편집기에서도 작성할 수 있습니다.
  • GitHub, GitLab, Bitbucket, Notion 일부 영역, VS Code 등에서 바로 미리보기가 됩니다.
  • HTML보다 짧고 읽기 쉬우며, 버전 관리에 적합합니다.
  • README, API 문서, 회의록, 블로그 초안, 학습 노트에 널리 쓰입니다.

작성 기본 규칙

  • 문단은 빈 줄 하나로 구분합니다.
  • 제목은 # 기호 개수로 단계가 정해집니다.
  • 목록, 코드블록, 인용문은 앞쪽 공백과 기호가 중요합니다.
  • 플랫폼마다 지원 문법이 조금 다를 수 있습니다.
핵심 감각: Markdown은 “보이는 모양”보다 “문서 구조”를 먼저 적는 방식입니다. 제목, 문단, 목록, 표처럼 의미를 명확히 잡으면 읽기도 좋고 관리도 쉬워집니다.

2. 제목과 문단

제목은 #을 사용합니다. #이 1개면 가장 큰 제목, 6개면 가장 작은 제목입니다. 일반적으로 문서에는 # 제목을 한 번만 쓰고, 아래 단계는 ##, ###처럼 순서대로 내려갑니다.

작성 코드 
# 문서 제목

## 큰 섹션

### 작은 섹션

본문 문단입니다. 문단과 문단 사이에는 빈 줄을 하나 둡니다.

새 문단입니다.
화면 결과

문서 제목

큰 섹션

작은 섹션

본문 문단입니다. 문단과 문단 사이에는 빈 줄을 하나 둡니다.

새 문단입니다.

주의: #제목처럼 # 뒤에 공백이 없으면 일부 뷰어에서 제목으로 인식하지 못합니다. 항상 # 제목처럼 공백을 넣는 습관을 권장합니다.

강조 문법 선택 기준

10. GitHub에서 유용한 확장 문법

GitHub Flavored Markdown, 줄여서 GFM은 기본 Markdown에 표, 체크박스, 취소선, 자동 링크, 이슈 참조 같은 기능을 더한 방식입니다. GitHub README나 Pull Request 설명을 작성한다면 아래 문법을 자주 쓰게 됩니다.

이슈와 PR 참조

#12처럼 작성하면 같은 저장소의 12번 이슈나 PR로 자동 연결됩니다.

owner/repo#34처럼 작성하면 다른 저장소의 이슈도 참조할 수 있습니다.

사용자 멘션

@username 형식으로 사용자를 언급할 수 있습니다. 협업 문서에서는 꼭 필요한 경우에만 사용합니다.

자동 링크

https://example.com처럼 URL을 그대로 적으면 자동으로 링크가 되는 경우가 많습니다.

11. README.md 작성 흐름

README는 프로젝트를 처음 보는 사람이 “무엇인지, 어떻게 실행하는지, 어디를 고치면 되는지” 빠르게 파악하도록 돕는 첫 문서입니다. 아래 순서대로 작성하면 대부분의 프로젝트에서 충분히 탄탄한 README를 만들 수 있습니다.

  1. 프로젝트 이름과 한 줄 소개
    문서 맨 위에는 프로젝트 이름을 # 제목으로 작성하고, 바로 아래에 핵심 설명을 한두 문장으로 적습니다.
  2. 주요 기능
    사용자가 얻는 기능을 목록으로 정리합니다. 구현 기술보다 사용자 관점의 결과를 먼저 쓰면 읽기 좋습니다.
  3. 설치와 실행 방법
    필요한 환경, 설치 명령어, 실행 명령어, 접속 주소를 코드블록으로 제공합니다.
  4. 폴더 구조
    중요한 파일과 폴더만 간단히 설명합니다. 모든 파일을 나열하면 오히려 읽기 어렵습니다.
  5. 사용 예시와 스크린샷
    CLI 도구라면 명령 예시, UI 프로젝트라면 화면 이미지를 넣습니다.
  6. 기여 방법과 라이선스
    협업 프로젝트라면 브랜치 전략, 커밋 규칙, PR 규칙, 라이선스를 적습니다.

12. 자주 하는 실수와 해결 방법

제목이 적용되지 않음

#제목처럼 붙여 쓰지 말고 # 제목처럼 공백을 넣습니다.

목록 안 코드블록이 깨짐

목록 아래 코드블록은 목록 항목보다 더 들여써야 합니다. 보통 공백 4칸 이상을 사용합니다.

표가 표로 보이지 않음

헤더 아래 구분 줄 | --- | --- |이 있는지 확인합니다. 파이프 개수도 맞춰야 합니다.

이미지가 보이지 않음

이미지 경로가 현재 Markdown 파일 기준으로 맞는지 확인합니다. 대소문자가 다른 경우도 실패할 수 있습니다.

줄바꿈이 원하는 대로 안 됨

Markdown은 일반 줄바꿈을 문단 안의 공백으로 처리할 수 있습니다. 확실히 줄을 바꾸려면 문장 끝에 공백 2칸을 넣거나 빈 줄로 문단을 나눕니다.

HTML과 Markdown이 섞여 이상함

HTML 블록 안에서는 Markdown이 해석되지 않는 경우가 있습니다. 복잡한 HTML은 별도 파일로 분리하는 편이 안전합니다.

14. 좋은 Markdown 문서 체크리스트

  • 문서 맨 위에 목적이 분명한 # 제목이 있습니다.
  • 제목 단계가 #, ##, ### 순서로 자연스럽게 내려갑니다.
  • 문단 사이에 빈 줄이 있어 원본 파일도 읽기 쉽습니다.
  • 명령어, 파일명, 변수명은 `인라인 코드`로 구분했습니다.
  • 긴 코드에는 언어 이름이 포함된 코드블록을 사용했습니다.
  • 이미지에는 의미 있는 대체 텍스트가 있습니다.
  • 표는 너무 넓지 않고, 모바일에서도 읽을 수 있을 만큼 간결합니다.
  • README에는 설치, 실행, 사용 예시, 폴더 구조가 포함되어 있습니다.
  • 링크가 실제로 열리는지 확인했습니다.
  • 최종적으로 미리보기 화면에서 깨지는 부분이 없는지 확인했습니다.

바로 써먹는 작성 순서

먼저 # 제목과 한 줄 설명을 쓰고, ## 제목으로 큰 섹션을 나눕니다. 그 다음 목록, 코드블록, 표, 링크를 필요한 곳에만 추가하면 깔끔한 Markdown 문서가 됩니다.
